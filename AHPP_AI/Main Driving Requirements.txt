PRODUCT REQUIREMENTS DOCUMENT
AI Racing Driver System for Live for Speed (LFS)

PURPOSE AND SCOPE

The AI Racing Driver System provides autonomous driving for Live for Speed vehicles using local InSim control and telemetry. The system drives AI vehicles along recorded routes, manages multi-lane traffic on looped routes, executes dynamic lane changes, handles non-looping branch routes, and maintains stable AI population levels in multiplayer environments.

Out of scope:

Machine learning or training systems

Physics replacement or override

Remote or cloud services

Primary objectives:

Stable highway-style autonomous driving

Continuous lane-based navigation on looped routes

Opportunistic and safe lane merging at runtime

Deterministic recovery from stalls or blockages

Clear in-game visualization and diagnostics

ROUTE SYSTEM

2.1 ROUTE CATEGORIES

The system must support the following route types:

Main Route
Loop: Yes
Required: Yes
Purpose: Primary driving lane

Main Alternate Route
Loop: Yes
Required: No
Purpose: Parallel second lane

Pit-to-Main Route
Loop: No
Required: Yes
Purpose: Spawn and pit entry

Branch or Detour Routes
Loop: No
Required: No
Purpose: Temporary deviations

AI must not spawn unless:

A valid Main Route exists

A valid Pit-to-Main Route exists

2.2 LOOP SEMANTICS

Main Route:

Must be a closed loop

First and last nodes are implicitly connected

AI may start at any node index

Navigation wraps seamlessly without reset

Main Alternate Route:

Must be a closed loop

Represents a parallel second lane of the Main Route

Shares direction and logical progression

Node indices do not need to align with Main Route

Both looped routes are always valid driving surfaces.

2.3 BRANCH ROUTE SEMANTICS (NON-LOOPS)

Branch routes:

Are linear and non-looping

Have exactly one attach point and one rejoin point

May attach to either Main or Main Alternate route

Rules:

AI may only enter a branch at its defined start

AI must exit at its defined end

Branch routes are one-directional and non-reversible

Lane changes are not permitted on branch routes

Failure to rejoin at the branch end must trigger recovery or despawn logic.

2.4 ROUTE DATA FORMAT

Routes are stored as JSON with ordered nodes.

Each node includes:

Position (X, Y)

Heading

Target speed

Optional per-node speed cap

Optional throttle and brake hints (non-authoritative)

Each route includes metadata:

Route type

Default speed limit

Weight or target ratio for AI allocation

Attach index (branches only)

Rejoin index (branches only)

Enabled flag

2.5 ROUTE VALIDATION

On route load:

Validate loop closure for Main and Main Alternate routes

Validate end-to-start distance and heading tolerance

Reject non-looped Main or Main Alternate routes

Reject looping branch routes

Validate attach and rejoin indices

Validate node spacing and curvature limits

Validation failures must:

Log to server console

Emit concise admin-only chat summaries

NAVIGATION AND PATH MANAGEMENT

3.1 BASE NAVIGATION

AI follows recorded routes using speed-scaled waypoint proximity thresholds

Waypoint progression is gated by heading alignment

Speed targets are derived from route and node metadata

3.2 LOOP NAVIGATION

Main and Main Alternate routes are traversed infinitely

No concept of start or end during runtime

AI may circulate indefinitely

LANE DETECTION AND MERGE SYSTEM

4.1 PARALLEL ROUTE DETECTION

The system must continuously detect parallel segments between the Main Route and Main Alternate Route.

Detection:

Occurs at runtime

Occurs at any point along the loop

Is not restricted to route boundaries

Parallelism criteria:

Lateral distance below threshold

Heading delta below threshold

Directional consistency

Parallel segments may exist intermittently.

4.2 LANE MERGE ELIGIBILITY

AI may initiate a lane merge at any point where:

Parallelism is detected

All merge safety rules are satisfied

There are no fixed merge nodes.

Lane merging:

Is bidirectional between Main and Main Alternate

May occur multiple times per lap

Is cooldown-limited

Is never permitted on branch routes

4.3 MERGE DECISION MODEL

Lane changes are:

Probabilistic

Cooldown-limited

Evaluated at fixed intervals

Safety checks include:

Parallel distance

Heading alignment

Speed delta

Clear merge corridor using MCI vehicle detection

4.4 SIGNALING AND TRANSITION PATH

Turn signals must activate before lateral movement

Minimum signal duration is enforced

Signals must deactivate only after merge completion

A temporary interpolation path must be generated:

Curvature-limited

Speed-aware

Followed until the merge completes

VEHICLE CONTROL AND DRIVING MODEL

5.1 STEERING

Primary controller is Pure Pursuit

Lookahead distance scales with speed

Steering damping and deadzone are configurable

Fallback:

Heading-error steering when pursuit is unstable

5.2 POWERTRAIN CONTROL

Human-like gear shifting

Explicit clutch timing

Stall detection and prevention

No instant gear snapping

5.3 SPEED CONTROL

Route and node-based speed targets

Proactive deceleration for curvature

Emergency braking overrides allowed

COLLISION AWARENESS AND RECOVERY

6.1 COLLISION DETECTION

Forward corridor vehicle detection using MCI data

Stop or slow when blocked

No overtaking unless on approved alternate route

6.2 RECOVERY STATE MACHINE

Recovery stages:

Pause and reassess

Reverse with steering bias

Forward retry

Escalation to pit or spectate

All recovery actions must be:

Time-boxed

Logged

Non-blocking

AI POPULATION MANAGEMENT

7.1 AUTO-POPULATION

AI count scales with human players

Minimum and maximum AI enforced

Reserved human slots respected

7.2 ROUTE ALLOCATION

AI distributed across routes using route weights

Allocation occurs only at spawn or despawn

No mid-route reassignment

All population changes must be thread-safe.

VISUALIZATION AND DEBUG TOOLING

8.1 IN-GAME UI

InSim panels must provide:

AI spawn and despawn controls

Auto-population toggle

Route recording controls

Visualization toggles

Live AI state display

8.2 LAYOUT VISUALIZATION

Routes rendered as layout objects

Sampling required to remain under object limits

Distinct colors per route type

Optional live waypoint markers

TELEMETRY AND PERFORMANCE

IS_MCI and IS_AII processed at or above 10 Hz

Control loop target under 100 ms

Rolling performance snapshots

Configurable logging windows

CONFIGURATION REQUIREMENTS

Configuration must expose:

AI population limits and ratios

Required route names

Lane merge probabilities and cooldowns

Steering and pursuit tuning

Recovery thresholds

Visualization sampling density

Hot reload should be supported where possible.

CONSTRAINTS AND LIMITATIONS

Must remain within LFS InSim packet limits

Must respect layout object caps

Must tolerate network latency

Must not block the main InSim thread

All behavior constrained by LFS physics
